# 条件分岐ユニット(cond) 問題設計概要

本ファイルでは、`cond_q01`〜`cond_q10` の問題内容・対応する知識要素・難易度と役割を整理し、
今後 `cond_q11`〜`cond_q14` をどう設計するかの方針と、他ユニット(loop/func)の問題数調整案をまとめる。

## 1. 既存問題一覧 (cond_q01〜cond_q10)

### cond_q01: 比較演算子と基礎概念
- **問題ファイル**: `data/mock_questions/cond_q01.json`
- **テキスト(text/prompt)**:
  - 「値が5と等しいか判定し、『xは5です』を表示するコードを作成してください。」
- **サンプル解答(sample_solution)**:
  - `x = 5` / `if x == 5:` / `print("xは5です")`
- **対応知識(requirements)**: `cond_q01_requirements.json`
  - 変数とは何か・代入の意味 (`x = 5`)
  - 比較演算子の種類と `==` による等値比較
  - `print()` による出力
  - インデント(ブロック)の役割
- **難易度/役割**:
  - 入門レベル。条件分岐というより「変数＋==＋print＋インデント」の最初の成功体験を与える問題。
  - 意図的誤答よりも「素直に正解を書いて合格」を優先する設計が適切。

### cond_q02: if-else文による条件分岐
- **問題ファイル**: `data/mock_questions/cond_q02.json`
- **テキスト**:
  - 「x > 5 なら 'big'、それ以外は 'small' を出力する if/else のコード。」
- **サンプル解答**:
  - `if x > 5:` / `print('big')` / `else:` / `print('small')`
- **対応知識**: `cond_q02_requirements.json`
  - `if 条件:` とインデントの基本構文
  - 条件が True/False のときの実行パス
  - `else:` の構文と「どちらか一方だけが実行される」という理解
- **難易度/役割**:
  - 初級。`if` だけから `if/else` に広げ、「二択の分岐」を体験させる問題。
  - ここもまずは正しく書ければ合格でよく、指摘学習は必須ではない。

### cond_q03: elifを使った3分岐分類
- **問題ファイル**: `data/mock_questions/cond_q03.json`
- **テキスト**:
  - `score` に応じて 'excellent' / 'good' / 'try again' を if/elif/else で出力する。
- **サンプル解答**:
  - `if score >= 80:` → 'excellent'
  - `elif score >= 50:` → 'good'
  - `else:` → 'try again'
- **対応知識**: `cond_q03_requirements.json`
  - `elif 条件:` の構文と if からの続き方
  - 上から順に条件を評価し、どれかが真になったら残りは実行しないという実行順序
  - 80以上/50以上80未満/それ以外、という「範囲の取り方」と else による残り全部の扱い
- **難易度/役割**:
  - 初〜中級。`if/elif/else` による三分岐の基本形を学ぶ問題。
  - requirements では「条件の順序を間違える」ことを first_attempt の典型誤りとして想定している。

### cond_q04: 短絡評価とゼロ除算回避
- **問題ファイル**: `data/mock_questions/cond_q04.json`
- **テキスト**:
  - 「短絡評価で、x != 0 のときだけ y/x を評価し、x == 0 のときは 'ゼロ除算を回避' を表示するコード。」
- **サンプル解答**:
  - `x != 0 and y / x > 0 and print(y / x)`
  - `x == 0 and print("ゼロ除算を回避")`
- **意図的誤答(intentional_mistake_template)**:
  - `x != 0` チェックを落としつつ y/x を評価してしまう危険なコード。
- **対応知識**: `cond_q04_requirements.json`
  - 論理演算子と短絡評価の仕組み
  - ゼロ除算の危険性と、その回避パターン
- **難易度/役割**:
  - 中級。「安全性・順序を考えた条件設計」を学ぶハイライト問題。
  - ここはマナビーが最初に意図的誤答を出し、ユーザに指摘させる指摘学習の中核として設計。

### cond_q05: notと論理演算子の組み合わせ
- **問題ファイル**: `data/mock_questions/cond_q05.json`
- **テキスト(text)**:
  - 「a>0 かつ b<10 の条件を not で否定し、条件を満たさないときに 'condition not met' を出力。」
- **サンプル解答**:
  - `if not (a > 0 and b < 10): print('condition not met')`
- **意図的誤答(intentional_mistake_template)**:
  - `if not a > 0 and b < 10:` のように括弧を忘れ、`not` が a>0 のみにかかる誤り。
- **対応知識**: `cond_q05_requirements.json`
  - `not` 演算子と否定の概念
  - 括弧によるグループ化と優先順位
  - `not` と `and` の組み合わせ
- **難易度/役割**:
  - 中級。`cond_q04` での論理・短絡の理解を前提に、「まとめて否定する」ときの括弧の重要性を学ぶ問題。
  - ここも意図的誤答＋指摘学習の核となる問題。

### cond_q06: grade関数による多段階条件分岐
- **問題ファイル**: `data/mock_questions/cond_q06.json`
- **テキスト**:
  - `grade(score)` を実装し、90以上A、70以上B、50以上C、それ以外Fを返す。
- **サンプル解答**:
  - if/elif/else で範囲を分け、`return` で文字を返す典型的な実装。
- **対応知識**: `cond_q06_requirements.json`
  - `elif` を複数使った多段階分岐
  - 条件を上から順に評価し、最初に当たったところで終わるという実行順序
  - `return` による関数からの値の返却
- **難易度/役割**:
  - 中級。`cond_q03` の3分岐を拡張し、より細かいレンジ分けと `return` の概念を結びつける問題。
  - requirements 上は「条件の順序を間違える」誤りを想定しており、将来指摘学習問題に拡張しやすい位置付け。

### cond_q07: andによる複数条件とmax3
- **問題ファイル**: `data/mock_questions/cond_q07.json`
- **テキスト**:
  - `max3(a,b,c)` を if/elif で実装し、3つのうち最大値を返す。
- **サンプル解答**:
  - `if a >= b and a >= c: return a` など、`and` を使って「aが最大か」を判定。
- **対応知識**: `cond_q07_requirements.json`
  - `and` による複数条件の同時判定
  - 最大値ロジックの考え方 (`a >= b and a >= c` 等)
  - if/elif/else による排他的な場合分け
- **難易度/役割**:
  - 中級。複数条件と「最大値を求める戦略」の両方を扱う応用問題。
  - first_attempt ミスとして「and の代わりに or を使う」ことを想定しており、論理演算の理解度チェックに向いている。

### cond_q08: インデントバグの修正
- **問題ファイル**: `data/mock_questions/cond_q08.json`
- **テキスト**:
  - インデントされていない `print('big')` を、`if x > 10:` の中に正しく入れるバグ修正問題。
- **サンプル解答**:
  - 日本語コメント＋修正済みの if/print インデント。
- **対応知識**: `cond_q08_requirements.json`
  - インデントとブロックの概念（主に構文レベル）
- **難易度/役割**:
  - 初〜中級。コードリーディングとシンタックスバグ修正にフォーカスした1問。
  - ここはユーザが直接バグを直す問題であり、マナビー側の意図的誤答を重ねる必要は低い。

### cond_q09: sign関数による符号判定
- **問題ファイル**: `data/mock_questions/cond_q09.json`
- **テキスト**:
  - `sign(x)` を実装し、正なら 'positive'、負なら 'negative'、ゼロなら 'zero' を返す。
- **サンプル解答**:
  - `if x > 0`, `elif x < 0`, `else` で三分岐＋`return`。
- **対応知識**: `cond_q09_requirements.json`
  - 符号判定ロジックと if/elif/else の組み合わせ
- **難易度/役割**:
  - 中級。`cond_q06` と同じく関数＋条件分岐だが、境界・レンジよりも「3ケースの網羅」に焦点。

### cond_q10: filter_positiveによる条件付き内包表記
- **問題ファイル**: `data/mock_questions/cond_q10.json`
- **テキスト**:
  - `filter_positive(nums)` を実装し、`x > 0` の要素だけを返すリスト内包表記を書く。
- **サンプル解答**:
  - `return [x for x in nums if x > 0]`
- **対応知識**: `cond_q10_requirements.json`
  - 条件付き内包表記と、`if` を使ったフィルタリングのイメージ
- **難易度/役割**:
  - 中〜やや上級（内包表記が登場するため）。
  - 条件分岐そのものというより「条件を用いたデータ処理」の一例として位置付け。

## 2. 知識内容・難易度・かぶりの評価

- **序盤( q01〜q03 )**:
  - q01: 変数・==・print・インデントの超入門。比較演算子全般を広く触れる。
  - q02: if/else の構文とTrue/False→実行パスの対応に集中。
  - q03: elif を導入し、多分岐と条件の順序・範囲設計を学ぶ。
  - → 役割分担は明確で、重複はむしろ意図的な「段階的な広がり」。

- **論理・短絡・否定( q04〜q05 )**:
  - q04: 短絡評価＋ゼロ除算回避という、安全な条件設計のハイライト。
  - q05: not＋括弧＋and の組み合わせを通じて、「条件の否定」とグループ化の重要性を学ぶ。
  - → テーマは密接に関連しているが、q04は "順序と安全性"、q05は "否定と括弧" と焦点が異なり、連続した2問として適切。

- **多段階分岐＋関数( q06〜q07, q09 )**:
  - q06: grade関数で「多段階レンジ＋return」を学ぶ。
  - q07: max3で「andによる複数条件＋最大値ロジック」を学ぶ。
  - q09: sign関数で「符号3ケースの網羅」と多分岐＋returnを再利用。
  - → どれも if/elif/else＋関数だが、
    - q06 はレンジ分け、
    - q07 は複数条件＋最大値、
    - q09 は符号3ケース、
    とテーマが少しずつ違うため、知識のかぶりは許容範囲。

- **構文バグ・内包表記( q08, q10 )**:
  - q08: インデントバグ修正に特化した「読む＋直す」問題。
  - q10: 内包表記と条件付きフィルタという発展的なスタイル紹介。
  - → 他の問題と直接の重複は少なく、バリエーションとして良い配置。

総じて、cond_q01〜q10 の範囲では「同じことを何度も別名でやっている」というレベルの無駄なかぶりはなく、
入門→基礎分岐→多分岐→論理演算・短絡→応用(関数・内包表記・バグ修正) という自然なスロープになっている。

## 3. cond を14問まで拡張する案

他ユニット(loop/func)を各14問に縮小したいという要望に合わせ、条件分岐(cond)を14問まで拡張する場合、
現在の10問の流れを崩さずに以下のテーマを追加するのが候補となる。

- **候補テーマ例 (q11〜q14)**
  - q11: 複数条件＋not の組み合わせ応用
    - 例: "年齢と会員ステータスから割引条件を判定" など、
      `not (条件1 or 条件2)` / De Morgan の法則に軽く触れる問題。
  - q12: 条件の早期リターンパターン
    - `if 条件: return ...` を複数並べて、残りを `return` でまとめるスタイル。
    - 可読性・バグの減りやすさを体感させる。
  - q13: 入れ子 if と論理演算の比較
    - 入れ子の if を `and` / `or` の1段階条件に書き換える／その逆を考える問題。
  - q14: 実務寄りの総合問題
    - ユーザの年齢・学年・テスト結果など複数情報を基に条件分岐し、メッセージを返す関数。
    - これまでの if/elif/else, and/or, not, 短絡, return を総動員させる総括問題。

実装にあたっては、既存の流れとの整合性をとるために、
- q11〜q12 を q06〜q07 の後あたりに挿入、q13〜q14 を最終応用として配置、
- `slot_index` と `qid` の対応、および progress 計算に用いる `total_questions` を合わせて更新、
といった調整が必要になる。

## 4. loop/func を14問に縮小する方針 (概略)

- ループ(loop)・関数(func)の問題は現在17問ずつ存在するが、
  cond を14問に拡張する関係で、loop/func はそれぞれ「基礎〜応用の代表14問」に整理する。
- 現段階では loop/func の requirements は作らず、
  cond 側の指摘学習ロジックが安定してから、同様のステージ設計を導入する方針とする。
- どの問題を削るか／統合するかは別途 loop_qXX / func_qXX を精査し、
  - 入門 (繰り返しの意味・for/while)
  - 典型パターン (合計・カウント・探索)
  - 応用 (ネスト・内包表記・再帰の導入 など)
  のバランスを見て決める。

## 5. 「マナビーに解かせる」ボタンの条件について

- 要望: **理解度が100%にならないと「マナビーに解かせる」を押せない仕組み**。
- 実装の方向性:
  - バックエンドとしては `learning_stage != 'CONCEPT_LEARNING'` かつ
    requirements 上の `completion_criteria` を満たしているときだけ、
    `can_attempt` フラグを True にするようにする、などの制御が考えられる。
  - フロントエンド側は `can_attempt` が False のときボタンを disable にし、
    「まずは教えを書く」「まだこの単元の理解を深めてね」といったメッセージを表示する。
- cond ユニットでは既に knowledge_items と completion_criteria が整っているため、
  ここを100% (all_checked) にすることを「理解度100%」の指標として扱える。
